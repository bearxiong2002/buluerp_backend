## 二、核心业务主线：订单生命周期

### 2.1 订单状态流转链

```
审核不通过 → 创建(未审核) → 待设计 → 设计中 → 待定制布产计划
→ 布产计划已定制(待排产) → 排产中 → 生产完成(待采购完成)
→ 已齐料入库(待套料) → 套料中 → 套料完成(待拉线)
→ 拉线组包中 → 拉线完成(待包装) → 包装中 → 已发货 → 已完成
```

### 2.2 状态流转关键特点

1. **字典驱动**：状态码不写死在Java枚举，而是通过`sys_dict_data`中`erp_order_status`字典动态查询。若字典缺失，全部状态流转失败。
2. **角色驱动**：`OrderStatus.STATUS_RULES`限制每个状态变更所需角色（design_dept、warehouse、wirestaying_dept、sell_dept、admin）。
3. **审核自动流转**：中间状态由布产/采购/包装审核结果自动触发，使用`OperationLog.OPERATOR_SYSTEM`以admin权限推进。

### 2.3 订单聚合关系

```
ErpOrders（订单）
  ├── ErpCustomers（客户）
  ├── ErpOrdersProduct（订单-产品关联）→ ErpProducts（产品）
  │                                      └── erp_product_material → ErpMaterialInfo（物料）
  ├── ErpDesignPatterns（设计总表）→ ErpDesignStyle（造型）→ ErpDesignSubPattern（子模式）
  │                                                      └── ErpMould（模具）→ ErpManufacturer（厂家）
  │                                                                        └── ErpMouldHouse（模具库）
  ├── ErpProductionSchedule（布产）→ ErpProductionArrange（排产）
  ├── ErpPurchaseCollection（采购汇总）→ ErpPurchaseInfo（外购资料）
  │                                    └── ErpPurchaseOrder（采购订单）→ ErpPurchaseOrderInvoice（发票）
  └── ErpPackagingList（包装清单）→ ErpPackagingBag（包装袋）→ ErpPackagingDetail（明细）
```

## 三、横向三链路

### 3.1 审核链路（ErpAuditRecordServiceImpl）

- **核心表**：`erp_audit_record`、`erp_audit_switch`
- **流程**：创建审核记录 → 发送通知 → 人工审核 → 回调业务Service推进状态
- **支持类型**：订单审核、布产审核、采购审核、包装审核
- **开关控制**：`IErpAuditSwitchService.isAuditEnabled(type)`，关闭时跳过审核直写终态

### 3.2 通知链路（ErpNotificationServiceImpl + NotificationAspect）

- **表**：`erp_notification`
- **机制**：审核创建时给目标角色发通知；`@MarkNotificationsAsRead`注解+SpEL自动标记已读
- **特点**：切面异常仅记录不阻断主流程

### 3.3 操作日志链路（OperationLogAspect + OperationLogInterceptor）

- **Controller层**：`OperationLogAspect`拦截带`@ApiOperation`的ERP Controller方法
- **持久层**：`OperationLogInterceptor`（MyBatis插件）拦截INSERT/UPDATE/DELETE，提取变更前后数据
- **注册状态**：**已确认通过`mybatis-config.xml`注册**，实际生效

## 四、关键问题诊断

### 4.1 安全风险 —— 严重

#### 问题1：ERP核心接口大面积匿名访问

- **现状**：ERP Controller中存在**204处`@Anonymous`注解**，大量核心接口（订单、客户、设计、物料、模具、布产、排产、采购、包装、库存等）被标记为匿名可访问
- **后果**：攻击者无需登录即可增删改查订单、客户、采购等核心业务数据
- **根因**：开发过程中为方便测试大量使用`@Anonymous`，`@PreAuthorize`被注释掉未恢复

**涉及Controller（全部方法均被标记@Anonymous）**：
- `ErpAuditRecordController` — 审核接口
- `ErpCustomersController` — 客户CRUD
- `ErpDesignPatternsController` — 设计总表
- `ErpDesignStyleController` — 设计造型
- `ErpMaterialInfoController` — 物料资料
- `ErpMaterialTypeController` — 物料类型
- `ErpManufacturerController` — 厂家管理

**修复建议**：
1. 全面清理ERP Controller上的`@Anonymous`
2. 恢复并统一`@PreAuthorize("@ss.hasPermi('system:xxx:xxx')")`
3. 在`sys_menu`和权限标识中补齐ERP权限
4. 仅保留登录、注册、验证码、Swagger等真正需要匿名的接口

#### 问题2：JWT密钥为弱密钥

- **位置**：`application.yml` 第79行
- **现状**：`secret: abcdefghijklmnopqrstuvwxyz`（连续字母）
- **后果**：JWT token可被轻易伪造，攻击者可构造任意用户身份token

**修复建议**：
1. 立即更换为随机生成的强密钥（至少256位）
2. 密钥移入环境变量或外部配置
3. 生产环境定期轮换

#### 问题3：数据库密码明文存储

- **位置**：`application-dev.yml`
- **现状**：
  - MySQL密码：`password: 060202`
  - Redis密码：`password: zU8YfMMXZXHtkK4QYfJr`
- **后果**：配置文件泄露即导致数据库完全暴露

#### 问题4：Druid监控控制台默认密码

- **位置**：`application-dev.yml` 第73-74行
- **现状**：`login-username: ruoyi`、`login-password: 123456`
- **后果**：Druid控制台可被轻易登录，暴露SQL执行、数据源连接等敏感信息

#### 问题5：Swagger生产环境未关闭

- **位置**：`application.yml` 第101行
- **现状**：`swagger.enabled: true`
- **后果**：生产环境暴露完整API文档，降低攻击门槛

### 4.2 前端缺失 —— 严重

#### 问题6：ERP前端页面和API封装完全缺失

- **现状**：
  - 后端有27个ERP Controller、完整的Service和Mapper
  - 前端`ruoyi-ui/src/api`只有若依原生API（login、menu、system等）
  - 前端`ruoyi-ui/src/views`只有若依原生页面，没有ERP页面
- **后果**：
  - 即使后端接口可用，前端无法调用ERP功能
  - 若`sys_menu`中配置了ERP菜单路径，前端路由懒加载会失败
- **缺失内容**：
  ```
  src/api/erp/orders.js
  src/api/erp/products.js
  src/api/erp/design.js
  src/api/erp/production.js
  src/api/erp/purchase.js
  src/api/erp/packaging.js
  src/api/erp/inventory.js
  src/api/erp/audit.js
  src/views/erp/...
  ```

**修复建议**：
1. 按业务主线补齐前端API封装
2. 开发ERP列表/详情/审核/导入导出/状态流转页面
3. `sys_menu`中配置菜单与Vue组件路径同步

### 4.4 配置与环境问题 —— 中等

#### 问题9：3D模型转换环境配置不匹配

- **位置**：`application.yml` 第15行
- **现状**：`python-path: ../tools/conda/envs/cadquery-env/bin/python`（Linux/Conda路径风格）
- **运行环境**：Windows本地
- **后果**：Windows下会回退到系统`python`，若未安装`cadquery`和`trimesh`则转换失败

**修复建议**：
1. 按环境区分配置（application-dev.yml用Windows路径，application-prod.yml用Linux路径）
2. 启动时验证Python环境和依赖
3. 转换失败原因返回给业务层

#### 问题10：Redis端口非标准

- **现状**：Redis配置端口为`46379`（非标准6379）
- **注意**：需确保部署环境Redis服务确实运行在该端口

---

### 4.5 代码质量问题 —— 中等

#### 问题11：权限注解与匿名注解混用

- **现状**：很多方法同时存在被注释掉的`@PreAuthorize`和生效的`@Anonymous`
- **问题**：权限设计痕迹存在，但实际未生效，代码维护混乱

#### 问题12：Service层循环依赖风险

- **现状**：多个ServiceImpl之间存在交叉注入（如ErpOrdersServiceImpl依赖IErpAuditRecordService，而ErpAuditRecordServiceImpl又反向调用IErpOrdersService）
- **风险**：Spring可能通过懒加载解决，但复杂场景下可能触发循环依赖异常

## 六、修复优先级建议

### 第一优先级（立即修复）

1. **清理`@Anonymous`**：移除所有ERP Controller的`@Anonymous`，恢复`@PreAuthorize`
2. **更换JWT密钥**：生成随机强密钥，移入环境变量
3. **加密数据库密码**：使用Jasypt或环境变量注入
4. **关闭生产Swagger**：`swagger.enabled: false`
5. **修改Druid密码**：生产环境禁用或强密码+白名单

### 第二优先级（本周内）

6. **补齐SQL初始化脚本**：将ERP业务表从dump整理为正式脚本
7. **开发ERP前端页面**：至少完成订单、客户、产品、审核等核心页面
8. **补齐前端API封装**：按业务模块封装axios调用

### 第三优先级（规划内）

9. **3D转换环境适配**：区分Windows/Linux配置
10. **清理未接入表**：确认`erp_bom`等表的业务含义，决定删除或接入
11. **Service解耦**：梳理并消除循环依赖风险

## 十、总结

本项目后端已形成以**订单为中心**、**审核通知为横向驱动**、**布产/采购/包装/库存为下游业务链**的ERP雏形，若依基础系统负责登录权限和菜单框架。但当前交付断点明确：

1. **安全风险是最大隐患**：204处`@Anonymous`使核心业务接口完全暴露，弱JWT密钥和明文密码进一步加剧风险
2. **前端是最大缺口**：后端接口完备但前端完全缺失，系统无法交付使用
3. **部署脚本是最大障碍**：31张ERP业务表不在标准初始化脚本中，新环境无法完整部署
4. **环境配置需要按实际收敛**：3D转换、数据库、Redis等配置需按真实部署环境调整

建议按"安全加固 → 补齐SQL → 补齐前端 → 功能迭代"的顺序推进。
