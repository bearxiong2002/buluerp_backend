## 一、项目概述

### 1.1 项目定位

本项目是基于 **若依 (RuoYi-Vue) 3.8.9** 框架二次开发的 **ERP（企业资源计划）管理系统**，服务于制造业（模具/注塑）场景，涵盖从订单、设计、布产、采购、排产、库存到包装出货的完整业务流程。

### 1.2 技术栈

| 层级 | 技术 |
|------|------|
| 后端框架 | Spring Boot 2.5.15 |
| 安全框架 | Spring Security 5.7.12 + JWT |
| ORM | MyBatis-Plus 3.5.1 + PageHelper |
| 数据库 | MySQL 8.x + Druid 1.2.23 连接池 |
| 缓存 | Redis (Lettuce) |
| API 文档 | Swagger 3 (Springfox) |
| 工具库 | Fastjson2, Apache POI, OSHI, Velocity |
| 前端 | Vue 2 + Element UI（ruoyi-ui 模块） |
| 辅助工具 | Python (cadquery + trimesh，STP→GLTF 转换) |

### 1.3 模块结构

```
buluerp_vue/
├── ruoyi-admin/        # 启动入口 + ERP 业务代码（Controller/Service/Mapper/Domain）
├── ruoyi-common/       # 通用工具类、注解、异常、基础实体
├── ruoyi-framework/    # 框架层（安全配置、过滤器、AOP、数据源）
├── ruoyi-system/       # 系统管理（用户/角色/菜单/部门/字典）
├── ruoyi-quartz/       # 定时任务
├── ruoyi-generator/    # 代码生成器
├── ruoyi-ui/           # Vue 前端
└── sql/                # 数据库初始化脚本
tools/
├── convert_stp_to_gltf.py  # 3D 模型格式转换
└── requirements.txt
```

### 2.1 订单生命周期

```
创建(未审核) → 待设计 → 设计中 → 待定制布产计划 → 布产计划已定制(待排产)
    → 排产中 → 生产完成(待采购完成) → 已齐料入库(待套料) → 套料中
    → 套料完成(待拉线) → 拉线组包中 → 拉线完成(待包装) → 包装中 → 已发货 → 已完成
```

状态流转由 `OrderStatus` 枚举中的 `STATUS_RULES` 控制，结合角色权限进行状态变更校验。

### 2.2 核心业务实体关系

```
ErpOrders (订单)
  ├── ErpOrdersProduct (订单-产品关联)
  │     └── ErpProducts (产品)
  │           └── 物料清单 (product_material 关联表)
  ├── ErpDesignPatterns (设计总表)
  ├── ErpProductionSchedule (布产计划)
  │     └── ErpProductionArrange (排产安排)
  ├── ErpPurchaseCollection (采购计划)
  │     └── ErpPurchaseOrder (采购订单) + ErpPurchaseOrderInvoice (发票)
  │     └── ErpPurchaseInfo (外购资料)
  ├── ErpPackagingList (包装清单)
  └── ErpAuditRecord (审核记录)
```

### 2.3 审核机制

系统通过 `ErpAuditSwitch` 控制各类审核的开关状态，支持以下审核类型：
- 订单审核
- 设计审核
- 布产审核
- 采购审核
- 排产审核
- 分包审核
- 外发审核

审核流程：创建审核记录 → 发送通知 → 审核人审批 → 通过/拒绝 → 触发后续业务逻辑。

## 三、发现的问题清单

### 🔴 严重问题 (Critical)

#### 3.1 所有 API 接口未做权限控制

**位置**：所有 `*Controller.java` 文件

**现象**：几乎所有 Controller 方法都使用了 `@Anonymous` 注解，且 `@PreAuthorize` 权限校验被注释掉。

```java
// ErpOrdersController.java 典型示例
@Anonymous  // ← 允许匿名访问
// @PreAuthorize("@ss.hasPermi('system:orders:list')")  // ← 权限校验被注释
@GetMapping("/list")
public TableDataInfo list(ListOrderRequest request) { ... }
```

**风险**：整个 ERP 系统的所有 API（订单、产品、采购、库存等）对外完全开放，任何人无需登录即可增删改查核心业务数据。这是**最严重的安全漏洞**。

**建议**：
- 移除所有 `@Anonymous` 注解
- 启用 `@PreAuthorize` 权限校验
- 仅对登录、注册、验证码等接口保留匿名访问

---

#### 3.2 数据库密码明文硬编码

**位置**：
- `application-druid.yml` 第 36 行
- `application-dev.yml` 第 34 行

```yaml
password: 1pReYZavtS5tZvyRfFxD  # 数据库密码明文
password: zU8YfMMXZXHtkK4QYfJr  # Redis 密码明文
```

**风险**：密码直接暴露在配置文件中，一旦代码泄露，数据库将面临直接攻击。

**建议**：
- 使用环境变量或外部配置中心（如 Nacos、Spring Cloud Config）
- 使用 Jasypt 等工具对敏感配置加密
- 将配置文件排除出版本控制

---

#### 3.3 数据库暴露在公网

**位置**：`application-druid.yml` 第 34 行

```yaml
url: jdbc:mysql://154.201.77.135:29783/buluerp?...
```

**风险**：数据库直接暴露在公网 IP 上，结合明文密码，攻击者可远程直接连接数据库。

**建议**：
- 数据库应仅监听内网地址
- 通过 VPN 或 SSH 隧道访问
- 使用防火墙限制来源 IP

---

#### 3.4 JWT 签名密钥过于简单

**位置**：`application.yml` 第 79 行

```yaml
token:
  secret: abcdefghijklmnopqrstuvwxyz
```

**风险**：密钥为连续字母，极易被暴力破解或字典攻击，导致 Token 可被伪造。

**建议**：
- 使用至少 256 位的随机字符串
- 通过环境变量注入，不写入配置文件
- 定期轮换密钥

---

#### 3.5 Druid 监控控制台弱密码

**位置**：`application-druid.yml` 第 75-76 行

```yaml
login-username: ruoyi
login-password: 123456
```

**风险**：Druid 控制台可查看 SQL 执行情况、数据源状态，弱密码使攻击者可获取敏感信息。

**建议**：修改为强密码，生产环境建议关闭或限制 IP 访问。

---

### 🟡 中等问题 (Major)

#### 3.6 N+1 查询问题

**位置**：
- `ErpOrdersServiceImpl.fillErpOrders()` — 循环查询产品信息
- `ErpPurchaseOrderServiceImpl.selectErpPurchaseOrderListByIds()` — 循环查询发票

```java
// ErpOrdersServiceImpl.java 第 68-77 行
for (ErpOrdersProduct erpOrdersProduct : products) {
    List<ErpProducts> erpProducts = erpProductsService.selectErpProductsListByIds(
        new Long[]{erpOrdersProduct.getProductId()}
    );
    // 每个产品一次 SQL 查询！
}
```

**风险**：数据量大时性能急剧下降。

**建议**：批量收集 ID，一次性查询后建立 Map 映射。

---

#### 3.7 重复依赖注入

**位置**：`ErpProductionScheduleServiceImpl.java` 第 49、58 行

```java
@Autowired
private IErpOrdersService erpOrdersService;  // 第 49 行

@Autowired
private IErpOrdersService erpOrderService;   // 第 58 行 — 同一个接口注入了两次！
```

**风险**：代码混乱，容易误用，且浪费内存。

**建议**：删除重复注入，统一使用一个变量名。

---

#### 3.8 事务边界不一致

**位置**：`ErpProductsController.importExcel()` 方法

```java
// 先校验所有行
for (AddProductRequest request : requests) {
    // 校验逻辑...
}
// 然后逐行插入，每行独立事务
for (AddProductRequest request : requests) {
    erpProductsService.processMaterialIds(request);
    erpProductsService.insertErpProducts(request);  // @Transactional
}
```

**风险**：如果第 5 行插入失败，前 4 行已提交无法回滚，导致数据不一致。

**建议**：将整个导入过程包裹在一个事务中，或使用编程式事务管理。

---

#### 3.9 代码重复

**位置**：
- `ErpProductsServiceImpl.parseActualPath()` (第 244 行)
- `ErpPurchaseOrderServiceImpl.parseActualPath()` (第 237 行)

两个完全相同的静态方法，功能是解析文件存储路径。

**建议**：提取到公共工具类（如 `FileUploadUtils` 或新建 `FilePathUtils`）。

---

#### 3.10 潜在空指针异常

**位置**：`ErpPurchaseOrderServiceImpl.deleteErpPurchaseOrderByIds()` 第 214-216 行

```java
String url = erpPurchaseOrderInvoice.getInvoiceUrl();
url = parseActualPath(url);  // 如果 url 为 null，parseActualPath 会抛 IllegalArgumentException
FileUtils.deleteFile(url);
```

**风险**：如果发票 URL 为空，删除操作会异常中断。

**建议**：添加 null 检查。

---

#### 3.11 配置文件路径错误

**位置**：`application.yml` 第 15 行

```yaml
python-path: ../tools/conda/envs/cadquery-env/bin/python
```

**风险**：这是 Linux 路径格式，在 Windows 环境下不存在。且 `tools/` 目录下实际没有 conda 环境。

**建议**：修正为 Windows 实际 Python 路径，或使用 `python` 命令依赖系统 PATH。

#### 3.12 配置文件不一致

**位置**：`application-dev.yml` vs `application-druid.yml`

| 配置项 | application-dev.yml | application-druid.yml |
|--------|---------------------|------------------------|
| 数据库地址 | localhost:29783 | 154.201.77.135:29783 |
| Redis 端口 | 46379 | 6379 |
| Redis 密码 | zU8YfMMXZXHtkK4QYfJr | (空) |
| 文件上传路径 | data/uploadPath | D:/ruoyi/uploadPath |

**风险**：两套配置差异巨大，切换 profile 时可能导致连接失败或数据写入不同位置。

**建议**：统一配置，使用环境变量区分不同部署环境。

### 🟢 轻微问题 (Minor)

#### 3.13 导出接口使用 POST 方法

**位置**：`ErpOrdersController.java` 第 74 行

```java
@PostMapping("/export")
public void export(HttpServletResponse response, Long[] ids) { ... }
```

**建议**：导出操作无副作用，建议使用 GET 方法，参数通过 Query String 传递。

---

#### 3.14 注释掉的代码过多

大量被注释的代码块散落在各 Service 中（如 `ErpOrdersServiceImpl` 中的设计总表查询、`ErpProductsServiceImpl` 中的物料关联插入），影响可读性。

**建议**：清理无用注释代码，依赖 Git 历史追溯。

---

#### 3.15 订单状态管理逻辑分散

订单状态变更逻辑散落在 `ErpOrdersService`、`ErpProductionScheduleService`、`ErpPurchaseCollectionService`、`ErpAuditRecordService` 等多个 Service 中。

**建议**：考虑引入状态机模式（如 Spring State Machine）统一管理订单状态流转。

---

#### 3.16 缺少输入校验

部分 Controller 接口（如 `ErpOrdersController.getInfo(Long[] ids)`）未对参数做空值校验。

**建议**：使用 `@Valid` / `@Validated` 注解进行参数校验。

---

#### 3.17 `ErpProductionScheduleServiceImpl.insertFromProduct()` 绕过了业务校验

`insertFromMaterial()` 通过调用 `insertErpProductionSchedule()` 间接触发了 `check()` 校验，但 `insertFromProduct()` 直接操作 Mapper 插入，绕过了 `checkUnique()` 和 `checkReferences()` 校验。

**建议**：统一通过 `insertErpProductionSchedule()` 插入，保持校验逻辑一致。

## 五、总结

本项目在业务层面设计较为完整，覆盖了制造业 ERP 的核心流程，审核机制、状态机设计也有一定深度。但在**安全性**方面存在严重缺陷——所有接口无权限控制、数据库密码明文、JWT 密钥过弱等问题叠加，使系统处于高风险状态。

**建议修复顺序**：
1. 立即修复安全问题（3.1 ~ 3.5）
2. 统一配置管理（3.12）
3. 优化性能和代码质量（3.6 ~ 3.10）
4. 逐步重构架构问题（3.15 ~ 3.17）
